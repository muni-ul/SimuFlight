# Feature 09 — Telemetry & Event Logging

> **Project:** AstraLoop — Python Software-in-the-Loop Flight Control & Validation System  
> **Feature:** Telemetry & Event Logging  
> **Document path:** `docs/features/09-telemetry-event-logging.md`  
> **Status:** Implementation specification  
> **Primary goal:** Capture one consistent, immutable, simulation-time-indexed record of AstraLoop's truth, measurements, mission state, requested commands, applied actuation, and active faults; collect sparse mission/fault/simulator events; and persist completed or diagnostic run bundles as inspectable CSV/JSON files without coupling simulation logic to plotting, validation, CLI presentation, or a database.

---

## Scope Boundary

**[Confirmed]** AstraLoop's selected architecture assigns the telemetry subsystem responsibility for:

- one consistent frame per simulation tick;
- CSV telemetry;
- event serialization;
- resolved configuration output;
- summary output;
- run-artifact organization.

**[Confirmed]** Every portfolio-ready run should ultimately produce an immutable local artifact directory containing:

```text
telemetry.csv
events.json
resolved_config.json
summary.json
flight_plot.png
```

The diagnostic plot itself belongs to Feature 11, but it is stored in the same final run bundle.

**[Confirmed]** Telemetry must include, at minimum:

```text
time
mission_state

true_x
true_y
true_vx
true_vy
true_theta
true_omega
true_mass

measured_x
measured_y
measured_vx
measured_vy
measured_theta
measured_omega

command_throttle
command_attitude
actual_throttle
actual_attitude

active_faults
```

**[Confirmed]** Mission transitions and fault lifecycle changes are generated by their owning subsystems and must be separately inspectable as events.

**[Confirmed]** The Simulation Engine records telemetry/events after a tick's physical advancement and invariant validation, before advancing the authoritative integer tick.

**[Confirmed]** Validation and plotting must remain separate from the running simulation:

- validation consumes completed run data;
- plotting consumes completed telemetry/run data;
- neither may mutate the live simulation.

**[Confirmed]** CSV and JSON are sufficient for the MVP. A database and Parquet are unnecessary.

**[Decision]** Feature 09 owns the **observability capture and file-artifact persistence layer**.

It owns:

- telemetry-frame data contract;
- telemetry schema version;
- event-envelope data contract;
- event schema version;
- in-memory `TelemetryRecorder`;
- one-frame-per-state-tick sequencing;

- frame/event validation;
- completed telemetry data object;
- CSV serialization;
- events JSON serialization;
- resolved-config JSON persistence;
- summary JSON persistence;
- immutable run-directory naming and creation;
- staging-directory and atomic-finalization behavior;
- diagnostic partial bundles for simulator failures where possible;

It does **not** own:

- objective validation formulas;
- PASS/FAIL outcome calculation;

- scenario TOML loading/resolution;
- campaign execution;
- a run-history database.

# 1. Feature Overview

## Feature name

**Telemetry & Event Logging**

---

## One-sentence description

**[Decision]** Implement a deterministic in-memory recorder and file-artifact writer that captures AstraLoop's tick-aligned system state and sparse lifecycle events, then serializes completed or diagnostic runs into immutable, human-readable CSV/JSON bundles.

---

## Detailed description

AstraLoop is a multi-layer stateful simulation:

```text
truth state
sensor measurements
measurement age/status
mission state
controller request
actuator response
fault state
simulation events
```

A reviewer must be able to inspect what happened **after the simulation has ended**.

The observability flow is:

```text
Simulation Engine tick
        |
        +--> truth state
        +--> MeasurementSnapshot
        +--> MissionStatus
        +--> ControllerUpdate
        +--> ActuatorUpdate
        +--> active fault IDs
        +--> mission/fault/simulator events
        |
        v
TelemetryRecorder
        |
        +--> immutable TelemetryFrame sequence
        +--> ordered RecordedEvent sequence
        |
        v
CompletedTelemetry
        |
        +--> Validator (Feature 10)
        +--> Plotter (Feature 11)
        +--> CSV/JSON serializers (Feature 09)
        |
        v
Immutable run artifact directory
```

The recorder is the one subsystem that owns the accumulated observability history.

It does not own the live state of any other subsystem.

---

## Core design principles

### 1. Simulation time, not wall-clock time

Telemetry timestamps use:

```python
sim_time = tick * dt
```

Wall-clock timestamps may appear only in run-directory metadata such as:

```text
run_started_at_utc
run_completed_at_utc
```

They never determine:

- frame order;
- event timing;
- fault timing;
- simulation behavior;
- validation.

---

### 2. Typed frames before serialization

The engine records a typed immutable `TelemetryFrame`.

It does not build arbitrary CSV dictionaries inside the simulation loop.

Serialization is a later boundary.

---

### 3. Sparse events remain separate from dense telemetry

Telemetry:

```text
one state-oriented frame per simulation state tick
```

Events:

```text
only meaningful lifecycle occurrences
```

Do not repeat every mission transition or fault event as free-form strings in every telemetry row.

---

### 4. Completed data is read-only

After recording finishes, the recorder returns immutable tuples:

```text
frames
events
```

Validator and plotter consume them without mutating the recorder.

---

### 5. Artifacts are logically immutable

Once a run directory is finalized:

- no file is appended to;
- no existing run is overwritten;
- a repeated run creates a new directory;
- the artifact bundle represents one completed attempt.

---

## Telemetry timing contract

Telemetry timing is one of the highest-risk cross-feature details.

The engine tick order is:

```text
A. terminal/safety check
B. fault update
C. sensor sampling
D. sensor fault/delay/stale behavior
E. MeasurementSnapshot
F. mission-state update
G. controller command
H. actuator update
I. RK4 truth advancement by dt
J. runtime invariant validation
K. telemetry/events recording
L. increment integer tick
```

A simple "record whatever is available at step K" can accidentally mix:

```text
truth at t + dt
```

with:

```text
measurements/commands generated at t
```

without documenting the relationship.

**[Decision]** AstraLoop uses a **state-tick frame model**.

---

## State-tick frame model

A telemetry frame is indexed by:

```text
tick = t
time = t * dt
```

and describes the system at the beginning of the control interval starting at that state tick.

A normal control frame contains:

- truth state at tick `t`;
- sensor snapshot produced from truth at tick `t`;
- mission state after the tick-`t` mission update;
- desired controller command for interval `[t, t+1]`;
- actuator state applied during interval `[t, t+1]`;
- active faults effective during interval `[t, t+1]`.

The engine then advances truth to tick `t+1`.

### Commit behavior

**[Decision]** The engine assembles the frame during the tick but only commits it to the recorder after:

- RK4 advancement succeeds;
- post-step invariants succeed.

This means every committed `CONTROL` frame corresponds to one successfully completed integration interval.

---

## Final terminal frame

After `N` successful steps, the final truth state exists at:

```text
tick = N
time = N * dt
```

No additional integration interval may begin because:

- mission is terminal;
- max time is reached;
- another normal termination reason occurred.

**[Decision]** Record one final:

```text
frame_kind = TERMINAL
```

at tick `N`.

It contains:

- final truth state;
- final mission state;
- latest available measurement snapshot;
- latest desired command;
- latest applied actuator state;
- latest active fault IDs;
- termination reason.

The latest software values are carried as last-known state and the `TERMINAL` frame kind makes clear that no new control interval was completed from that row.

### Resulting row count

For a run with:

```text
N successful integration steps
```

normal telemetry contains:

```text
N CONTROL frames
+ 1 TERMINAL frame
```

Therefore:

```text
frame_count = completed_steps + 1
```

This gives plots both the initial state and final endpoint.

---

## Terminal-at-start run

If the engine is terminal at tick `0` before any integration step:

```text
one TERMINAL frame at tick 0
```

No CONTROL frame is committed.

---

## Simulation error frame

If a simulator/runtime error occurs during tick `t` after the tick's software decisions exist but before successful state commitment:

**[Decision]** Record one best-effort:

```text
frame_kind = ERROR
```

at tick `t`.

It contains:

- last valid committed truth state;
- current/latest software-visible values if available;
- attempted command/applied actuator state if available;
- error code/message in the frame metadata;
- no claim that interval `[t, t+1]` completed.

The recorder must never invent a next state.

The diagnostic bundle policy is described later.

---

## Frame kinds

Recommended enum:

```python
class TelemetryFrameKind(Enum):
    CONTROL = "control"
    TERMINAL = "terminal"
    ERROR = "error"
```

Do not add many frame types without a real lifecycle need.

---

## One frame per state tick

**[Decision]** The recorder accepts at most one frame for each integer tick.

A normal successful run therefore contains ticks:

```text
0, 1, 2, ..., N
```

without gaps or duplicates.

The final tick is TERMINAL.

An error run contains the valid/control sequence followed by an ERROR frame at the next/current state tick.

---

## Why record state tick rather than only completed-interval number

It aligns:

- initial state at tick 0;
- sensor timestamps;
- mission transitions;
- fault events;
- final state;
- plotting x-axis;
- validation metrics.

The command recorded at tick `t` is explicitly the command used for the interval beginning at tick `t`.

---

## Telemetry frame content

Recommended in-memory shape:

```python
@dataclass(frozen=True)
class TelemetryFrame:
    schema_version: int
    kind: TelemetryFrameKind

    tick: int
    time: float

    mission_state: MissionState

    truth: VehicleState
    measurement: MeasurementSnapshot | None

    controller: ControllerUpdate | None
    actuator: ActuatorUpdate | None

    active_fault_ids: tuple[str, ...]

    termination_reason: str | None = None
    error_code: str | None = None
    error_message: str | None = None
```

The frame contains typed domain records.

The CSV serializer flattens them.

---

## Why use nested typed records in memory

It avoids a giant constructor with dozens of loosely related fields.

It also preserves subsystem boundaries:

```text
VehicleState
MeasurementSnapshot
ControllerUpdate
ActuatorUpdate
```

The serializer owns column naming.

## Simulation lifecycle events

**[Decision]** Feature 09/engine should also support sparse simulator events:

```text
simulation_started

```

### `simulation_started`

Occurs at:

```text

```

Includes:

- scenario ID;

- config digest.

### `simulation_terminated`

Occurs at final terminal tick.

- termination reason;

### `simulation_error`

## Unified event envelope

Mission and fault modules should retain their typed domain events.

The recorder wraps them in one stable serialized envelope.

Recommended:

```python
@dataclass(frozen=True)
class RecordedEvent:
    event_schema_version: int
    sequence: int

    tick: int
    time: float

    source: EventSource
    event_type: str
    code: str
    message: str

    data: Mapping[str, JsonScalar | JsonArray | JsonObject]
```

```python
class EventSource(Enum):
    SIMULATION = "simulation"
    MISSION = "mission"
    FAULT = "fault"
```

Optional controller/sensor events should not be added until a real sparse-event requirement exists.

## Event schema version

**[Decision]**

```text
event schema version = 1
```

`events.json` contains it at the file level and each event may omit the repeated field if the file-level schema is authoritative.

Recommended file shape:

```json
{
  "event_schema_version": 1,
  "scenario_id": "nominal",
  "seed": 42,
  "config_digest": "full-sha256",
  "events": [
    {
      "sequence": 0,

      "message": "Simulation started.",

    }

```

---

## Event sequence

Events can share the same tick and simulation time.

**[Decision]** The recorder assigns a strictly increasing integer:

```text
sequence = 0, 1, 2, ...
```

in the exact order events are received from the engine.

This preserves total ordering.

---

## Same-tick event ordering

The engine's subsystem order determines event order.

Recommended same-tick order:

```text
1. fault deactivation events
2. fault activation events
3. mission transition event
4. simulation termination/error event
```

`simulation_started` is first at tick 0.

This matches the actual tick lifecycle.

The recorder must not arbitrarily sort same-tick events by message or ID.

---

## Event validation

Every recorded event must have:

- nonnegative tick;
- finite simulation time equal to `tick * dt`;
- nonempty type;
- nonempty code;
- nonempty message;
- JSON-safe structured data;
- sequence matching recorder's next sequence.

No NaN/Inf values may exist in event data.

---

## Significant events only

Do not use events.json as verbose debug logging.

Do not record:

- every sensor sample;
- every controller update;
- every actuator update;
- every telemetry frame;
- every valid mission guard check.

Those belong in telemetry or verbose standard logging.

---

## Standard-library logging vs events

Use standard `logging` for developer-facing runtime diagnostics.

Use `events.json` for structured domain/lifecycle events.

A message printed to console is not a substitute for an event record.

An event record is not automatically printed to console.

Feature 15 decides presentation.

---

## Finalization

Calling:

```python
finalize()
```

returns:

```python
@dataclass(frozen=True)
class CompletedTelemetry:
    telemetry_schema_version: int
    event_schema_version: int

    dt: float
    frames: tuple[TelemetryFrame, ...]
    events: tuple[RecordedEvent, ...]
```

After finalization:

- recorder cannot accept more frames;
- recorder cannot accept more events;
- repeated `finalize()` may return the same immutable object or raise.

**[Decision]** Repeated finalization returns the same `CompletedTelemetry`.

This makes cleanup code idempotent.

---

## Frame sequencing rules

### First frame

Allowed first tick:

```text
0
```

### Subsequent frame

Must be:

```text
previous_tick + 1
```

### Final frame

Must be:

```text
TERMINAL
```

or:

```text
ERROR
```

No frames may follow.

### CONTROL frame

May not be the final object after recorder finalization unless an error in caller prevented proper final frame creation.

**Decision:** `finalize()` rejects a nonempty sequence ending in `CONTROL`.

This catches incomplete recording.

### Empty recorder

**Decision:** `finalize()` rejects no-frame runs.

Every started simulation attempt must have at least a TERMINAL or ERROR frame.

A configuration failure occurs before recorder/run start and has no telemetry.

---

## Frame validation

At recording boundary:

- schema version known;
- tick nonnegative;
- time finite;
- `time == tick * dt` within strict timestamp tolerance;
- truth values finite;
- mission state valid;
- active fault IDs stable/unique;
- measurement values either finite or `None`;
- measurement status/value combinations consistent;
- controller values finite when present;
- actuator values finite when present;
- CONTROL frames have controller/actuator data required by the configured runtime;
- TERMINAL/ERROR frames may contain last-known/optional data;
- error metadata appears only for ERROR frame;
- termination reason appears on TERMINAL/ERROR frame.

---

## Measurement consistency checks

Examples:

### VALID/STALE reading

Normally requires:

```text
value is finite
source_tick is not None
age_ticks is not None
```

### UNAVAILABLE/DISABLED

Normally:

```text
value is None
```

Feature 03 is the main owner of these invariants, but telemetry should reject internally inconsistent snapshots rather than serialize misleading data.

---

## Non-finite values

**[Decision]** Telemetry never serializes:

```text
NaN
+Inf
-Inf
```

Engine/runtime boundaries should catch invalid state first.

If non-finite data reaches the recorder:

```text
raise TelemetryError
```

Do not silently write `"nan"`.

For a SimulationError caused by non-finite state, record an ERROR frame using the **last valid state** and error metadata.

---

## Memory model

At the expected MVP scale:

```text
dt ≈ 0.02 s
max_time ≈ 60 s
frames ≈ 3001
```

A list of typed frames is small enough for in-memory capture.

**[Decision]** Do not stream telemetry to disk during the MVP run.

### Why

- simpler atomic bundle creation;
- completed immutable data available to validator/plotter;
- no partially written normal CSV;
- easy tests;
- no file handle in simulation loop.

Future larger campaigns may add streaming/Parquet separately.

## CSV serialization

Recommended API:

```python
def write_telemetry_csv(
    path: Path,
    telemetry: CompletedTelemetry,

    ...
```

### Encoding

```text
UTF-8
```

### Newlines

Open with:

```python
newline=""
```

Use:

```text
\n
```

as lineterminator for stable cross-platform output where practical.

### Header

Use a centralized immutable:

```python
TELEMETRY_COLUMNS: tuple[str, ...]
```

Do not derive field order from dictionary insertion by accident.

### Missing values

Serialize `None` as empty field.

Do not serialize:

```text

"null"
"N/A"
0
```

### Boolean values

Serialize as lowercase:

```text

false
```

or stable numeric `1/0`.

**Decision]** Use lowercase `true`/`false` for readability.

### Enums

Serialize stable explicit enum values.

### Floats

Use one documented round-trip-safe formatting helper.

Recommended:

```python
repr(float_value)
```

under pinned Python 3.13, after finite validation and signed-zero normalization where appropriate.

Do not arbitrarily round telemetry to presentation precision.

The plot/CLI can format values later.

### Active faults

Use compact JSON array text in the CSV cell.

## Artifact write failure

Introduce:

```python
class ArtifactError(Exception):
    ...
```

or a domain-specific equivalent.

Examples:

- root cannot be created;
- disk write fails;
- rename fails;
- serializer cannot encode supposedly valid data;
- plot file missing during required finalization.

**[Decision]** Artifact failure is not mission validation failure.

The writer should:

- preserve the original exception as cause;
- clean up the staging directory on a best-effort basis;
- never publish a normal-looking incomplete final directory;

## Event and telemetry relation

A telemetry frame at tick `t` and an event at tick `t` share the same simulation timestamp.

Examples:

- fault activates at tick `t`;
- `active_faults` in CONTROL frame `t` includes the fault;
- `events.json` contains activation event at tick `t`;
- measurement/controller/actuator fields in frame `t` show the resulting current-tick behavior.

Mission transition:

- transition event at tick `t`;
- frame `t` contains the new mission state;
- controller command in frame `t` uses the new profile.

This relationship must be covered by integration tests.

## Recorder pseudocode

```python
class TelemetryRecorder:
    def __init__(self, *, dt: float) -> None:
        self._dt = dt
        self._frames = []
        self._events = []
        self._last_frame_tick = None
        self._next_event_sequence = 0
        self._completed = None

    def record_frame(self, frame: TelemetryFrame) -> None:
        if self._completed is not None:
            raise TelemetryError("recorder finalized")

        validate_frame(frame, dt=self._dt)

        expected_tick = (
            0
            if self._last_frame_tick is None
            else self._last_frame_tick + 1
        )

        if frame.tick != expected_tick:
            raise TelemetryError(...)

        if (
            self._frames
            and self._frames[-1].kind
            in {TERMINAL, ERROR}
        ):
            raise TelemetryError(...)

        self._frames.append(frame)
        self._last_frame_tick = frame.tick

    def record_events(
        self,
        events: Iterable[DomainEvent],
    ) -> None:
        for event in events:
            recorded = adapt_event(
                event,
                sequence=self._next_event_sequence,
                dt=self._dt,
            )
            validate_recorded_event(recorded)
            self._events.append(recorded)
            self._next_event_sequence += 1

    def finalize(self) -> CompletedTelemetry:
        if self._completed is not None:
            return self._completed

        if not self._frames:
            raise TelemetryError(...)

        if self._frames[-1].kind is CONTROL:
            raise TelemetryError(...)

        self._completed = CompletedTelemetry(
            dt=self._dt,
            frames=tuple(self._frames),
            events=tuple(self._events),
            ...
        )
        return self._completed
```

## Engine integration pseudocode

```python
tick = 0
truth = initial_state

record(simulation_started_event(tick=0))

while True:
    if terminal_before_cycle(...):
        record_terminal_frame(
            tick=tick,
            truth=truth,
            latest_context=...,
        )
        record(simulation_terminated_event(...))
        break

    tick_context = execute_discrete_software_for_tick(
        tick=tick,
        truth=truth,
    )

    frame = build_control_frame(
        tick=tick,
        truth=truth,
        context=tick_context,
    )

    try:
        next_truth = advance_truth(...)
        validate_runtime_invariants(next_truth)
    except SimulationError as exc:
        recorder.record_frame(
            build_error_frame(
                tick=tick,
                truth=truth,
                context=tick_context,
                error=exc,
            )
        )
        recorder.record_event(
            simulation_error_event(...)
        )
        raise

    recorder.record_frame(frame)
    recorder.record_events(
        tick_context.events
    )

    truth = next_truth
    tick += 1
```

At the next loop, final truth is represented by TERMINAL if terminal.

Event-record timing relative to frame append may vary internally, but final event sequence must reflect engine order.

# 6. Acceptance Criteria

## AC-01 — Recorder starts empty

**Given** a new recorder  
**When** inspected internally through supported state/result behavior  
**Then** it contains no frames/events and is not finalized.

---

## AC-02 — First frame must be tick zero

**Given** an empty recorder  
**When** frame tick 1 is recorded  
**Then** `TelemetryError` is raised.

---

## AC-03 — Tick zero frame is accepted

**Given** a valid tick-zero CONTROL/TERMINAL/ERROR frame  
**When** recorded  
**Then** it becomes the first frame.

---

## AC-04 — Frames must be sequential

**Given** last frame tick `t`  
**When** frame tick `t+1` is recorded  
**Then** it is accepted.

---

## AC-05 — Duplicate frame tick is rejected

**Given** tick `t` already recorded  
**When** another tick `t` frame is recorded  
**Then** it is rejected.

---

## AC-06 — Skipped frame tick is rejected

**Given** last tick `t`  
**When** tick `t+2` is recorded  
**Then** it is rejected.

---

## AC-07 — Frame time equals tick times dt

**Given** frame tick `125` and `dt=0.02`  
**When** validation runs  
**Then** time must equal `2.5` within strict timestamp tolerance.

---

## AC-08 — Inconsistent frame time is rejected

**Given** tick/time mismatch  
**When** recording is attempted

## How this feature helps the project stand out

Telemetry turns AstraLoop's behavior into evidence.

A strong claim is:

> "Every run produces a tick-aligned record of simulation truth, software-visible measurements, mission mode, requested control, actual actuator response, and active faults, plus a separate ordered event trace and resolved configuration."

This demonstrates:

- engineering observability;
- reproducibility;
- debugging discipline;
- serialization design;
- testability;
- file-system artifact management.

---

## What to mention in README

Recommended wording:

> **Structured run artifacts:** Each persisted run creates an immutable local bundle containing tick-aligned CSV telemetry, ordered mission/fault events, the fully resolved configuration and seed, objective summary data, and a diagnostic plot.

Useful bullets:

- one frame per simulation state tick;
- truth and measurements stored separately;
- measurement status/age;
- requested vs actual actuation;
- active fault IDs;
- exact transition/fault events;
- no database;
- same run analyzable after completion.

## What to mention in interviews

### How do you avoid off-by-one telemetry?

> "I defined a state-tick frame contract. Frame t contains truth and software decisions for the interval beginning at tick t. It is committed only after that interval successfully advances. A final terminal frame records the endpoint at tick N."

### Why separate events from telemetry?

> "Telemetry is dense and regular; events are sparse and semantic. Mission transitions and fault lifecycle events retain exact ordering without repeating strings on every CSV row."

### How do you align a fault event with sensor behavior?

> "The fault manager runs before sensor sampling. An activation event at tick t and telemetry frame t both show the fault active, and the frame's measurement/control values reflect the same-tick effect."

### Why no database?

> "A run is naturally an immutable artifact bundle, not relational application data. CSV and JSON are easier to inspect, delete, share and reproduce, with no migration or service overhead."

### How do you avoid partial artifacts?

> "The writer builds a sibling staging directory and only renames it to the final timestamp-plus-config-digest path after all required files are complete."

### What happens on a simulator error?

> "The simulator error remains an error, but I best-effort save an error frame, event, partial telemetry, resolved config and diagnostic summary so the failure is debuggable."

### How do you keep serialization reproducible?

> "Column order is fixed, enums and booleans have stable encodings, JSON rejects NaN, event order has sequence numbers, and wall-clock metadata is excluded from telemetry/events/config."

### Does telemetry calculate PASS/FAIL?

> "No. The recorder captures facts. The validator evaluates completed data, and the summary serializer stores that result."

# Open Questions

1. **[Open Question] Should PID P/I/D internals be included in telemetry schema version 1?**  
   Recommended: omit until controller tuning demonstrates a real need.

2. **[Open Question] Should measurement age seconds be stored in addition to age ticks?**  
   Recommended: no; derive using resolved `dt`.

3. **[Open Question] Should terminal frame carry last-known measurement/command values or leave them empty?**  
   Recommended: carry latest values and use `frame_kind=terminal` to make semantics explicit.

4. **[Open Question] Should SimulationError diagnostic bundles attempt plot generation?**  
   Recommended: only if Feature 11 can safely plot partial data; plot is not required for error bundle.

5. **[Open Question] Should summary include Git commit/version/platform metadata?**  
   Include only through small best-effort logic; never require Git/network.

6. **[Open Question] Should run directories use eight or twelve config-digest characters?**  
   Eight is readable for local scale; full digest remains in JSON.

7. **[Open Question] Should CSV float formatting use Python `repr` or a documented `.17g` helper?**  
   Choose one, test round-trip behavior, and freeze it before stable artifact examples are published.

8. **[Open Question] Should a separate `artifact_manifest.json` exist?**  
   Recommended: no. Put relative artifact names in summary.json.

9. **[Open Question] Should telemetry include controller target setpoints per row?**  
   Mission state plus resolved profiles allow reconstruction, but explicit targets improve debugging. Add only if the controller exposes them cleanly.

10. **[Open Question] Should active faults also be represented as individual boolean columns?**  
    Recommended: no. JSON-array cell scales better and events provide lifecycle details.

11. **[Open Question] Should final normal bundle publication require plot success?**  
    Portfolio-ready CLI should require the five-file bundle; lower-level tests may persist data-only bundles before Feature 11.

12. **[Open Question] Should staging directories be removed or preserved after artifact failure?**  
    Recommended: remove best-effort; preserve hidden staging only when cleanup fails and report path.

13. **[Open Question] Should wall-clock run timestamps be captured by Feature 08 runner or Feature 09 artifact writer?**  
    Recommended: runner provides start/end UTC metadata; writer only serializes/uses it.

14. **[Open Question] Should summary JSON include full final truth state as well as derived metrics?**  
    Recommended: yes, because it helps debugging without opening CSV.

15. **[Open Question] Should events.json include configured faults that never activated?**  
    Do not create fake lifecycle events. Summary should list pending/unactivated configured fault IDs.
